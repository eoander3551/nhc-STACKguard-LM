SG-1 — LM Website Modules

No Collector Required

SG-1 uses LM's Website monitoring modules to check reachability of a customer IP from LM's own infrastructure. The device is not onboarded as an LM Resource. No Collector is deployed. This is already working. The Website module is retired when a site upgrades to SG-2+.

SG-2 and Above — LM Resources

Collector + Cloud Controller

Devices are onboarded as LM Resources with a deployed NHC Collector. Reachability is derived from the cloud controller for supported platforms (VeloCloud, Bigleaf, UniFi, Peplink, Cradlepoint) or from LM Collector ICMP ping (PingMulti DataSource) for non-controller platforms (Juniper, Fortinet, Cisco). nhc.sg_sku controls what is surfaced to the customer and what actions are enabled.

01 Purpose

NHC's STACKguard service currently uses LogicMonitor for ICMP reachability monitoring with auto ticket creation in CustomCall — this is SG-1 and it works. This document specifies everything NHC needs LM to do across the full STACKguard product line.

Key Architectural Distinction SG-1 uses LM's Website monitoring modules — LM checks a customer IP from its own infrastructure. No Collector is deployed and the device is not onboarded as an LM Resource. SG-2 and above move to LM Resources: devices are onboarded with a Collector, DataSources run against them, and the full alert and remediation stack operates at the Resource layer. These are two distinct LM constructs and the build approach for each is different.

For SG-2 through SG-4, LM is configured once to support all capabilities across all managed sites. A property on each Resource (nhc.sg_sku) determines what gets ticketed, what is displayed in the customer portal, and whether remediation fires. The SKU is the product — LM is the engine.

Multi-WAN Alert Dependency & Suppression

Without suppression a single site outage generates a device ICMP alert plus individual alerts for every WAN circuit and interface simultaneously. The following parent-child rules must be configured for all managed sites.

Core Principle When a device is unreachable (ICMP down), all child DataSource alerts on that device are suppressed. One composite alert fires. Rules below are additive and work together.

Rule 1 — All WANs Down + Device Unreachable → Site Down

Condition: icmp.reachable = 0 AND cloud controller confirms all WAN circuits down
Action: Fire single CRITICAL: Site Down — suppress all child WAN and interface alerts
Ticket: P1 — CustomCall NOC Immediate

Rule 2 — Single WAN Down, Device Reachable

Condition: icmp.reachable = 1 AND one WAN circuit = down per cloud controller
Action: Fire CRITICAL: WAN Circuit Down on the specific circuit only. Note in ticket that failover may be active.
Ticket: P2 — CustomCall NOC Urgent

Rule 3 — WAN Degraded, Device Reachable

Condition: icmp.reachable = 1 AND circuit metric threshold exceeded
Action: Fire WARNING: WAN Circuit Degraded on the specific circuit
Ticket: P3 — CustomCall NOC Monitor

Rule 4 — Controller Reports Circuit Down, ICMP Reachable

Condition: Cloud controller (VCO, Bigleaf) reports circuit = down, device ICMP = reachable
Action: Fire CRITICAL: WAN Circuit Down — tag alert source as controller name. This pattern distinguishes an ISP-side circuit failure from a CPE device failure.

06 Automated Remediation

LM is configured to support remediation workflows for all sites. Whether remediation actually fires at a given site is gated by nhc.sg_sku and nhc.pdu_auto_reboot_enabled. The initial workflow is PDU-triggered circuit recovery.

Safeguards Required Before Production Enablement Cooldown enforcement, SDT checks, and full audit logging must all be in place and validated in dry-run mode before nhc.pdu_auto_reboot_enabled is set to true on any production site.

Workflow: WAN Circuit Down → PDU Port Reboot

All conditions must be true to fire

  • Alert type = WAN Circuit Down (Critical)
  • Alert active for ≥ nhc.pdu_trigger_delay_min minutes (default: 3)
  • nhc.pdu_present = true and nhc.pdu_auto_reboot_enabled = true
  • No active Scheduled Downtime on the device or device group
  • No reboot performed on this outlet within nhc.pdu_cooldown_min minutes (default: 60)
1
WAN Circuit Down alert fires in LM after 2-cycle confirmation
↓
2
Escalation chain webhook fires to NHC Automation Endpoint with site_id, device_id, circuit_instance, alert_id. CustomCall ticket created by alert rule at this point if not already open.
↓
CHK
Automation validates all conditions — reads device properties from LM API, checks SDT, checks cooldown log. If any condition fails, automation logs the reason and exits without action — ticket note added: "Auto-remediation skipped: {reason} (SDT active / cooldown not elapsed / PDU not enabled)"
↓
3
Resolve PDU target — reads nhc.pdu_ip and nhc.pdu_port_wan1/2 from device properties based on circuit instance
↓
4
Call Synaccess PDU REST API — POST /restapi/powerCtrl → reboot outlet. Log timestamp. Ticket note added: "Auto-remediation initiated: PDU outlet {N} reboot triggered at {time} for {circuit_instance}. Monitoring for recovery."
↓
5
Write LM Event to device — "Auto-remediation: PDU outlet {N} rebooted — WAN circuit recovery attempt" (secondary audit trail in LM)
↓
CHK
Monitor for recovery over 5-minute window via LM alert state or cloud controller poll
↓
6A
Recovered: Write LM Event confirming recovery. Ticket updated: "Circuit {instance} recovered at {time} following PDU reboot. No further action required." Ticket resolved/closed.
6B
Not recovered: Write LM Event. Ticket updated: "Auto-remediation attempted — PDU outlet {N} rebooted at {time}, no circuit recovery detected after {N} minutes. Manual intervention required." Ticket priority escalated to P1, assigned to NOC queue.

NOC Ticket Context Requirements

When a CustomCall ticket is created, the support team must be able to pull a context snapshot without logging into LM directly:

  • Current ICMP status + 15-min latency and loss trend
  • WAN circuit status table — per circuit: status, latency, loss, utilization
  • Cloud controller site summary where applicable
  • Device health snapshot — CPU, memory, temperature
  • Last 5 site alerts with timestamps and resolution status
  • PDU reboot history (last 7 days) if nhc.pdu_present = true
  • Active SDTs at this site

10 Platform Implementation Notes

Ubiquiti UniFi

  • UniFi Controller REST API preferred over SNMP
  • UDM Pro WAN interface state = authoritative circuit source
  • Device ICMP to UDM is parent for all site child alerts
  • Confirm LM Exchange DS against current UniFi API schema

Ubiquiti UISP

  • UISP REST API only — no reliable SNMP for CPE metrics
  • Key: link quality, CCQ, signal, throughput per station
  • Custom Groovy DataSource required
  • Map UISP device via nhc.controller_site_id

Juniper EX2300

  • SNMP: JUNIPER-ALARM-MIB + IF-MIB
  • Key: port status, error counters, chassis temp
  • SNMP traps → LM EventSource for link events
  • No cloud controller — SNMP is authoritative

VMware VeloCloud

  • VCO REST API: getEnterpriseEdgeList, getEdgeStatusMetrics
  • Per-link: latency, loss, jitter, bandwidth, quality score
  • VCO webhook → LM EventSource for link state changes
  • Dual-path: VCO circuit state + device ICMP for suppression
  • Map edge via nhc.controller_site_id

Bigleaf SD-WAN

  • Bigleaf Cloud REST API — circuit status, quality tier, failover
  • Custom Groovy DataSource required — no Exchange DS
  • Webhook → LM EventSource for circuit state changes
  • Dual-path: Bigleaf API + device ICMP for suppression
  • Map site via nhc.controller_site_id

Fortinet FortiGate

  • SNMP: fnSysCpuUsage, fnSysMemUsage, fgTempSensor, IF-MIB
  • FortiOS REST API: SD-WAN SLA, VPN tunnels, HA state
  • SNMP traps: link up/down, HA failover → LM EventSource
  • Verify Exchange DS covers SD-WAN SLA datapoints

Cisco 2400 (VoIP)

  • SNMP: CISCO-VOICE-DIAL-CONTROL-MIB for calls, MOS, jitter
  • CISCO-PROCESS-MIB (CPU), CISCO-MEMORY-POOL-MIB (memory)
  • IF-MIB for WAN utilization and error counters
  • Confirm Exchange DS covers VoIP MIBs

Peplink B1

  • InControl2 API preferred for WAN health + cellular metrics
  • SNMP pepwave MIB as fallback
  • Key: per-SIM RSRP, RSRQ, SINR, carrier, connection state
  • Custom DS likely needed for InControl2 — confirm Exchange

Cradlepoint 1850

  • NetCloud API preferred over SNMP for full cellular data
  • Key: WAN zone state, RSRP/RSRQ/SINR, usage per interface
  • Map via nhc.controller_site_id = NetCloud router ID
  • Confirm Exchange DS availability for NetCloud API

Synaccess PDU

  • ICMP monitored as standard device (SG-1 baseline)
  • Outlet state polled via REST API for health visibility
  • REST API called by NHC automation for SG-4 reboot actions
  • All actions logged as LM Events on the associated edge device

11 Build Phases

Phases represent the order in which the LM platform is built out — not product tiers. All capabilities are built into a single unified LM configuration; SKU-gating happens at the NHC application layer.

Phase 1 — SG-1 Website Module Setup (Complete)
  • LM Website modules configured per customer IP
  • Check interval and alert confirmation cycles set
  • Alert Rules → CustomCall ticket creation on unreachable
  • Auto-clear on recovery
  • No Collector deployed, no Resource onboarding required
Phase 2 — Resource Onboarding & Circuit Monitoring
  • LM Collector deployment and validation at NHC PoPs
  • Device onboarding as LM Resources with property schema applied
  • Exchange DS gap analysis across full fleet
  • API DataSources: VeloCloud, Bigleaf, Peplink, Cradlepoint, UISP
  • WAN circuit metric schema across all platforms
  • Parent-child alert dependency / multi-WAN suppression
  • Cloud controller EventSource ingestion (VCO + Bigleaf webhooks)
  • Enriched CustomCall ticket payload with circuit context
Phase 3 — Device Health & Normalization
  • Device health DataSources: CPU, memory, temp, interface errors
  • Cellular metric DS for Peplink and Cradlepoint
  • VoIP metric DS for Cisco 2400
  • LM API endpoint validation for NHC portal integration
  • NHC standard site dashboard template in LM
  • SLA report configuration per site / per circuit
Phase 4 — Automated Remediation
  • Remediation device property schema on PDU-enabled sites
  • Escalation chain webhook for circuit-down trigger
  • NHC automation service: PDU API, cooldown, LM event write
  • Recovery detection and CustomCall ticket update
  • Dry-run / audit mode before production opt-in per site
  • Extensibility framework for future remediation actions
